# Спецификация нормализатора — MedArchive (ТЗ 4.3, вес 25%)

Задача: сопоставить «сырое» название услуги из прайса (`price_item.service_name_raw`)
с записью целевого справочника (`service`). Результат — `service_id` + `match_confidence`.
Цель MVP по ТЗ: **≥70% позиций сопоставляются автоматически**; реалистичная планка
с эмбеддингами — 85%+.

Это отдельный модуль `normalizer.py`. Стыкуется с бэкендом через `repository.py`
(см. раздел «Интерфейс»).

---

## 1. Предобработка текста (до сопоставления)
Даёт больше прироста, чем выбор модели. Поверх `utils.text.normalize_key`:

- **NFKC + lowercase + ё→е** (уже есть в `normalize_key`).
- **Расшифровка аббревиатур** по словарю: ОАК → «общий анализ крови», ОАМ, УЗИ, ЭКГ,
  ЭхоКГ, КТ, МРТ, РЭГ, ФГДС и т.д. Словарь — отдельный `abbreviations.json`, пополняемый.
- **Лемматизация** (pymorphy3): снять падежи/числа — «крови»→«кровь»,
  «консультации»→«консультация».
- **Удаление довесков**: категория сложности («1 кат.»), единицы («1 ед.», «шт»),
  дозировки, коды источника, скобочные примечания, ценовые суффиксы.
- **Стоп-слова**: «услуга», «прием», «исследование», «анализ» (как самостоятельные —
  аккуратно, не выкидывать значимые).
- **Гомоглифы** латиница/кириллица (частично уже в `utils/prices.py`).

Выход: `normalized_name` — каноничная строка для сравнения.

---

## 2. Каскад сопоставления (precision-first)
Идти по ступеням, останавливаться на первой уверенной:

1. **Точное совпадение** нормализованного ключа со `service.service_name`.
2. **Синонимы**: совпадение с элементами `service.synonyms`.
3. **Аббревиатура → ключ** после расшифровки.
4. **Лексический fuzzy** — RapidFuzz `token_set_ratio` / `WRatio`
   (устойчив к перестановке слов).
5. **Семантика** — эмбеддинги (см. §3) для перефразировок.

`match_confidence` = нормализованный скор ступени, на которой нашли (0..1).

---

## 3. Семантические эмбеддинги
RapidFuzz не поймёт, что «развёрнутый анализ крови» ≈ «общий анализ крови».

- Модель: многоязычная `sentence-transformers` —
  `paraphrase-multilingual-MiniLM-L12-v2` или **LaBSE** (лучше для русского+казахского).
- Предпосчитать эмбеддинги справочника один раз, держать в памяти/кэше.
- Гибридный скор: `0.5 * лексика + 0.5 * косинус` (веса калибруются).

---

## 4. Пороги (два, не один)
- Скор ≥ `AUTO_THRESHOLD` (напр. 0.90) → автосопоставление: пишем `service_id` + `match_confidence`.
- Скор ≤ `UNMATCHED_THRESHOLD` (напр. 0.70) → `service_id = NULL` (очередь `/unmatched`).
- Между → очередь ручной верификации (флаг на ревью).
- Пороги — в конфиге, калибруются на размеченной выборке.

---

## 5. Сужение кандидатов (blocking)
Не сравнивать со всем справочником:
- сначала кандидаты через `pg_trgm` (индекс `idx_service_name_trgm` уже есть)
  или по `category` (лаборатория / диагностика / консультация / процедура);
- затем переранжировать эмбеддингами.
Быстрее и точнее — не даёт матчить анализ крови на консультацию.

---

## 6. Доменные ловушки (различать как разные услуги)
- правый / левый;
- с контрастом / без контраста;
- 1 / 2 / 3 триместр, категория сложности;
- первичный / повторный приём;
- взрослый / детский.

---

## 7. Оценочный стенд (обязательно)
Без него «хорошо» нельзя измерить.
- Gold-набор: 200–300 размеченных пар «raw → service_id».
- Метрики: precision / recall / accuracy автосопоставления.
- Итеративно крутить пороги, предобработку, веса гибрида.
- Артефакт: `eval/normalizer_eval.py` + отчёт.

---

## 8. Обратная связь
Подтверждения/исправления оператора (`POST /match`, `POST /verify`)
писать обратно в `service.synonyms` — система улучшается со временем.

---

## Интерфейс (стыковка с бэкендом)
Бэкенд (мой) даёт `repository.py` и схему БД. Нормализатор не лезет в БД напрямую
мимо репозитория.

**Вариант A — на этапе загрузки (рекомендуется):**
```python
# normalizer.match(raw_name, category=None) -> MatchResult
@dataclass
class MatchResult:
    service_id: str | None
    confidence: float | None   # 0..1, None если не сопоставлено
    stage: str                 # exact | synonym | abbrev | fuzzy | semantic | none
```
Загрузчик вызывает `normalizer.match(...)` до вставки `PriceItem`,
проставляет `service_id` / `match_confidence`.

**Вариант B — пост-обработка уже загруженных:**
```python
# normalizer.run_over_unmatched(repository) — проходит price_item где service_id IS NULL,
# обновляет через repository.update_match(item_id, service_id, confidence)
```

**Что предоставляет бэкенд:**
- `repository.load_services()` → список услуг справочника (имя + синонимы + категория);
- `repository.update_match(item_id, service_id, confidence)`;
- индекс `idx_service_name_trgm` (есть) для blocking;
- эндпоинты `/unmatched` и `/match` (читают/пишут `service_id`).

**Что предоставляет нормализатор:**
- модуль `normalizer.py` с `match()` / `run_over_unmatched()`;
- словари `abbreviations.json`, стоп-слов;
- оценочный стенд и отчёт по точности.

---

## Зависимости
```
rapidfuzz
sentence-transformers   # эмбеддинги (LaBSE / multilingual-MiniLM)
pymorphy3               # лемматизация русского
```

## Приоритет реализации
1. Предобработка + ступени 1–4 (точное/синонимы/аббрев/fuzzy) — быстрый путь к ~70%.
2. Оценочный стенд — чтобы видеть прогресс.
3. Эмбеддинги (ступень 5) + калибровка порогов — путь к 85%+.
