# Моя часть (бэкенд)

## Этап 0. Привести модели в соответствие со схемой
**Зачем:** `PriceItem` (dataclass) не совпадает с таблицей `price_item` — без этого
загрузчик не соберётся корректно.

- В `models.PriceItem` добавить `service_id: str | None = None`,
  `match_confidence: Decimal | None = None` (эти поля заполняет нормализатор — чужая часть,
  но колонки в БД мои).
- `source_row` / `source_sheet` — служебные, в БД не пишутся (вынести из payload вставки).
- В `PriceDocument` добавить `file_path: str | None`, согласовать `created_at`.
- Единый маппинг «поле модели → колонка БД» в одном месте (`repository.py`).

**Готово, когда:** payload вставки содержит ровно колонки таблиц.

## Этап 1. Загрузчик в Supabase (repository-слой)  ← начинаем отсюда
**Зачем:** связать готовый парсер с базой; наполнение базы — часть оценки.

- Модуль `repository.py` поверх `db.get_cursor()`.
- Вставки с `ON CONFLICT`:
  - `upsert_partner(partner)` — дедуп по `bin`, иначе `name`+`city`.
  - `insert_document(document)` → `doc_id`.
  - `bulk_insert_items(items)` — пакетная вставка с учётом `uq_item_dedup`.
- Транзакция на документ: партнёр → документ → позиции; ошибка → откат, статус `error`.
- Идемпотентность: повторная загрузка того же файла не плодит дубли.
- Точка входа: `ArchiveParser.parse()` → `Repository.save(payload)`;
  в `cli.py` флаг `--to-db`.

**Готово, когда:** после прогона архива в Supabase есть строки в
`partner` / `price_document` / `price_item`, повторный прогон не дублирует.

## Этап 2. REST API на FastAPI (ТЗ 4.5, вес 15%)
**Зачем:** `main.py` в README заявлен, но не существует; нужен поиск и Swagger.

- `main.py` — FastAPI, зависимость БД из `db.get_cursor`.
- Эндпоинты:
  - `GET /services` — фильтр по категории.
  - `GET /services/{id}/partners` — кто оказывает услугу + цены резидент/нерезидент.
  - `GET /partners` — фильтр по городу/статусу.
  - `GET /partners/{id}/services` — весь прайс партнёра.
  - `GET /search?q=` — нечёткий поиск (`pg_trgm`, индексы уже есть).
  - `GET /unmatched` — очередь несопоставленных (читает `service_id IS NULL`).
  - `POST /match` — ручное сопоставление (пишет `service_id`).
  - `POST /items` — приём позиции от парсера.
  - `GET /stats` — метрики дашборда (запрос отдаёт команда этапа 5, я делаю роут-каркас).
  - `GET /health` — проверка БД.
- Pydantic-схемы ответов; `GET /docs` (Swagger) — автоматически.

**Готово, когда:** Swagger открывается, выборки и поиск возвращают данные из Supabase.

> API я строю так, чтобы поля `service_id` / `match_confidence` уже были в схеме ответа.
> Их наполняет нормализатор (чужая часть) — мне достаточно их читать.
